# Execute Tech Spec

Write a detailed, comprehensive technical specification of the current project by performing a fresh analysis of the entire codebase.

## Execution policy

This skill follows `../EXECUTION_POLICY.md`. In particular: delegate the codebase analysis to a fresh subagent (per the "Fresh context" rule), and follow the failure-handling rules if writing `techspec.md` fails.

## Overwrite behavior

Write directly to `techspec.md` in the project root. If it already exists, overwrite it without asking. Source control is the safety net for recovering the prior version.

## Document what IS, not what SHOULD BE

This is a descriptive specification, not a review. Do NOT identify flaws, suggest improvements, flag anti-patterns, or recommend changes. Only document what actually exists in the codebase today. Design flaws are the concern of `execute-techreview`.

## Analysis process

**Delegate the analysis phase to a fresh subagent.** Spawn a subagent (via the Agent tool) whose sole job is to survey the codebase and return the structured findings needed to fill each section of the template below. This ensures the analysis is performed in a genuinely fresh context, unpolluted by the main session.

After the subagent returns its findings, the main skill writes `techspec.md` from those findings following the output rules below.

The subagent's task:

1. **Survey the project** — Use Glob, Grep, Read, and Bash to thoroughly explore:
   - Project root files (package.json, Cargo.toml, go.mod, *.csproj, Makefile, Dockerfile, etc.)
   - Directory structure and organization
   - Source code across all modules
   - Configuration files and environment setup
   - Test structure and patterns
   - Build and deployment files
   - Documentation files

2. **Review prompt and Claude files** — Read `project.prompt.md`, `CLAUDE.md`, and files under `.claude/` to understand intended patterns and strategies. However, only include information in the techspec if it can be **confirmed as actually implemented in the codebase**. Do not document aspirational or planned-but-unbuilt items.

3. **Analyze and categorize findings** into the sections defined below.

4. **Evaluate for reproducibility** — For each section, ask: "Is this enough information for an AI with no other context to regenerate this aspect of the project to a substantially equivalent result?" If not, add more detail. The techspec is the authoritative document for recreating the project — it must be complete enough to do so without access to the source code.

## Output format

Write the techspec to `techspec.md` in the project root using the following structure. Use clean markdown. Keep each section self-contained so that diffs are localized when individual sections change.

When writing the techspec, follow these rules:
- **Be specific.** Use actual file names, actual class names, actual function names. Reference real paths.
- **Be comprehensive.** Cover the full codebase, not just the parts that seem most important.
- **Be flat.** Avoid deeply nested structures. Prefer sections and subsections (max 3 levels) so diffs stay readable.
- **One topic per paragraph.** This keeps source control diffs localized — a change to one concept does not cause adjacent unrelated lines to shift.
- **Use lists and tables** where they improve scannability.
- **Omit empty sections.** If a section has no content (e.g., no database in the project), omit it entirely rather than writing "N/A".
- **Reference, don't define.** When the project uses well-known protocols, standards, or patterns (e.g., REST, OAuth 2.0, MVC, WebSocket), name them but do not explain them. Assume the reader knows what they are. Only describe project-specific usage or configuration choices.
- **No opinions.** Do not editorialize, critique, or suggest. Describe only.

Use the following section structure as the template. Omit any section that has no content rather than writing "N/A".
